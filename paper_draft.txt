## Training Objective

The goal of training is to learn a useful internal description of short audio and seismic clips before we ever ask the model to label a vehicle. We call this internal description the latent representation. The labels we care about are whether a vehicle is present, what type it is, how close it is, and what the background looks like. Instead of training one big classifier from raw signal to label, we first train an encoder to organize the signal into named blocks of latent variables, and only then train small probe heads on top. This split lets us reuse the same backbone for many tasks, and it lets us check that the right kind of information is landing in the right block.

[FIGURE 1: High-level CRL concept — paired audio and seismic windows pass through a frontend and encoder into a structured latent, with downstream probes on top. Path: crl-train/saved_crl/slides-figs/fig12_crl_concept.png]

The framework supports three training objectives, and each one is paired with a different loss. The first is a variational autoencoder, or VAE, objective. The total VAE loss is

    L_vae = L_recon + β · L_kl + λ_interv · L_interv + λ_pres · L_aux_pres + λ_type · L_aux_type,

where the reconstruction term is mean-squared error on the frontend features, L_recon = MSE(x_hat, x), and the KL term keeps the latent close to a standard normal prior:

    L_kl = (1/B) Σ_b  (1/2) Σ_d  ( exp(log_var_b,d) + μ_b,d^2 − 1 − log_var_b,d ).

The intervention-matching term is a binary cross-entropy over a two-bit target (presence-changed, type-changed) computed between paired windows, L_interv = BCEWithLogits([s_pres, s_type], [y_pres, y_type]). The auxiliary heads use weighted cross-entropy by default, and a focal variant when class imbalance is severe:

    L_focal = ( Σ_i  w_yi · (1 − p_yi)^γ · ( − log p_yi ) ) / Σ_i w_yi.

The classification terms use class weights so rare vehicle types still pull the gradient, and they read the posterior mean μ rather than a noisy sample, so the routing of information into blocks is not blurred. The second objective is a disentangled VAE that splits the latent into a signal block and an environment block and adds three extra L2 losses,

    L_align  = mean_b ‖ μ_signal,audio − μ_signal,seismic ‖^2,
    L_stable = mean_{b ∈ consec} ‖ μ_env,t − μ_env,t+1 ‖^2,
    L_invar  = mean_b ‖ μ_signal,clean − μ_signal,intervened ‖^2,

so that the audio and seismic signal estimates agree, the environment block changes slowly across nearby windows, and the signal block is invariant to noise interventions. The third objective is contrastive learning with the NT-Xent loss over stratified partners,

    L_ntxent = mean_b  − (1 / |P_b|)  Σ_{p ∈ P_b}  log  exp(sim(a_b, k_p)/τ) / Σ_k exp(sim(a_b, k)/τ),

where P_b is the set of positive partners for anchor b (consecutive-window or same-type), all other partners in the batch act as negatives, and sim is the cosine similarity between L2-normalized embeddings.

[FIGURE 2: Training flow for the three CRL modes. The shared encoder feeds a structured latent z, and three branches — VAE, disentangled, and contrastive — apply mode-specific losses and select their own checkpoint metric. Path: crl-train/saved_crl/slides-figs/fig9_training_flow.png]

These objectives were chosen because each one targets a real weakness of a simpler approach. A plain supervised classifier can learn shortcuts that work in one location but fail in another, since it has no pressure to separate signal from background. The VAE objective adds that pressure through reconstruction and the KL term, and the auxiliary heads keep the representation tied to the labels we care about. The disentangled objective goes one step further by writing the signal-versus-environment split into the loss itself, which is important when the model has to generalize to new sensor sites. The contrastive objective is included because some downstream probes do better with a deterministic, sharply separated representation than with a probabilistic one, and NT-Xent gives that without needing a decoder. Having all three modes share the same encoder and the same data pipeline lets us compare them fairly. The contrastive mode is implemented and shares the same encoder, but it has not been trained or evaluated yet, so the results in this paper come from the VAE and disentangled objectives only. The contrastive path is described here as part of the framework, not as a measured comparison.

A practical concern shaped one more piece of the loss design. The KL term in the VAE is multiplied by a coefficient β that is annealed during training, so the model is not punished for using the latent too early. That makes the raw training loss hard to compare across epochs, because the same number means different things at different β values. To pick the best checkpoint we therefore evaluate a β-independent reference ELBO at β = 1,

    val_ref_elbo = val_recon + val_raw_kl,

and we save a second checkpoint based on the auxiliary type-F1 score, so we have both a generative and a discriminative anchor. Early stopping watches val_ref_elbo. This design keeps the training loss free to anneal while the model-selection metric stays honest.

## Datasets

The model is trained on two multi-sensor datasets, both collected by outside groups. The first is the MOD_vehicle dataset, recorded by a joint team from the Army Research Laboratory (ARL) and the University of Illinois Urbana-Champaign using Raspberry Shake nodes during their field tests. MOD_vehicle is split across two collection sites, FOCAL and IOBT, that share one sensor setup and one file format but were recorded at different locations with different vehicle fleets. The FOCAL site holds short field recordings of pedestrians, bicycles, scooters, motorcycles, sedans like a Subaru Forester and a Ford Mustang, and pickups like a Tesla Model 3 and a generic pickup truck. The IOBT site adds two military-style light vehicles (a Polaris MRZR and an AM General Warhog) and a Chevrolet Silverado pickup, recorded on test ranges with longer continuous passes. We treat FOCAL and IOBT as separate site labels inside MOD_vehicle so that a model can be evaluated on a held-out site for honest cross-location numbers, but the underlying data pipeline is one and the same. The second dataset is M3NVC, an open research dataset that contributes background scenes and four passenger vehicles (Mazda CX-30, Mazda Miata, Ford Mustang, and Mercedes-Benz GLE 350) recorded with a different sensor rig. Each recording session, regardless of source, captures synchronized audio and seismic streams from a node placed near the vehicle path. Audio is captured at 16 kHz on MOD_vehicle (both sites) and at 1.6 kHz on M3NVC; seismic is captured at 100 Hz on MOD_vehicle and at 200 Hz on M3NVC. To make the streams comparable, audio is always resampled to 16 kHz at load time and seismic to 100 Hz, so every window the model sees has the same shape regardless of which dataset or site it came from.

[FIGURE 4: One paired (audio, seismic) window from IOBT (Silverado pass), de-meaned per channel as the model pipeline applies. Three full-height kernel-window boxes are overlaid on each panel to show the multiscale receptive fields at three different sizes; kernel sizes are exaggerated relative to the real defaults so the receptive fields read at print scale. Path: crl-train/saved_crl/slides-figs/fig4_window_examples.png]

The raw streams are sliced into non-overlapping 1-second windows. Each window holds 16,000 audio samples and 100 seismic samples, and is paired with a single presence label (vehicle in scene, yes or no) computed by majority vote over the per-sample presence flags inside that window, plus a 4-class vehicle-type label drawn from {pedestrian, light, medium, heavy}. Trailing samples that do not fill a complete window are dropped. The full evaluation split contains about 91,325 paired windows, and the leaderboard runs report on this same 91,325-window full split so numbers are directly comparable across runs. Class weights are computed per split so that rare types still pull gradient during training; absent classes get weight zero rather than a large inverse-frequency value, which keeps the loss scale stable when a split happens to miss a class. The CRL training run uses raw waveforms — no spectrogram is precomputed — so the multiscale frontend learns its filters directly from the time-domain signal.

[TABLE 2: Per-dataset window counts, sensor rates, and vehicle classes. Source: crl-train/crl_vehicle/config.py (DATASET_VEHICLE_MAP) and crl-train/saved_crl/runs/multiscale/*/eval/.../full/eval_report.json (n_windows). Path: rendered inline below this section in a future revision.]

The train, validation, and test splits are assigned at the recording level, not at the window level, so no individual recording session ever appears in two splits. For FOCAL and IOBT, each (vehicle, session) entry in the dataset map carries an explicit marker — "train", "val", "test", or "split". A "split" marker means the recording is cut in half: the first half goes to validation and the second half to test, in window-index order, so adjacent windows within the same recording stay on the same side of the split. For M3NVC, which has long multi-pass recordings, a "split_runs" marker is used: each (scene, run) inside the parquet file is treated as a unit, paired across audio and seismic so both sensors agree on the boundary, and then assigned to train / val / test by a deterministic 50 / 25 / 25 greedy partition that balances paired-window counts across the three buckets. A built-in floor guarantees that, when at least three runs survive the pairing step, validation and test each receive at least one run; otherwise the smallest train run is swapped into the empty bucket. This recording-level discipline matters for honesty: a window-level random split would let the model see the same vehicle pass in both train and test, which inflates results without measuring real generalization.

The pretraining pipeline uses two augmentation mechanisms, both of which are part of the training objective rather than a one-time preprocessing step. The first is stratified partner sampling: every anchor window is paired with one partner from each of four strata — a consecutive window from the same recording, a different recording with the same vehicle type, a different vehicle type, and a different dataset entirely — so the encoder sees, for free, examples that share or differ in specific ways. The second is intervention noise. Seven generators (white, brown, pink, green band-limited, low-frequency sinusoid, high-frequency chirp, and bird-chirp-like band-limited noise) are each available; one is sampled per anchor at 20 percent RMS of the signal and added to the input. The disentangled training mode uses these interventions to drive its signal-invariance loss, which pushes nuisance variation into the environment block. Per-window DC removal is applied once before the frontend so a constant offset cannot bias the learned filters. No spectrogram-style augmentations (time masking, frequency masking, mixup) are used.

## Training Hyperparameters

The two reported runs (Multiscale + VAE and Multiscale + Disentangled) share an identical hyperparameter set; the only knob that differs between them is `training_mode`, which selects which loss the framework assembles. Holding everything else fixed lets the comparison in this paper attribute any gap in performance to the training objective rather than to a coupled change. The following values come straight from each run's `meta.json` config block; both runs were trained on a single GPU.

The optimizer is AdamW with an initial learning rate of 3 × 10⁻⁴ and a minimum learning rate of 1 × 10⁻⁴, decayed across all 100 epochs by a cosine annealing schedule. Weight decay is 1 × 10⁻⁴. There is no dropout anywhere in the encoder or decoder; the transformer is built with `dropout=0.0`. The model relies on three forms of regularization instead of dropout: the KL term in the VAE loss, which acts as a Bayesian regularizer on the latent posterior; the auxiliary classification heads, which keep the representation tied to labels; and the partner-pair losses (intervention matching for VAE, alignment / temporal-stability / signal-invariance for disentangled), which constrain the geometry of the latent without explicit weight noise. The KL coefficient β starts at 0 and is annealed in steps of 0.02 toward 1, with adaptive logic that lifts β when the raw KL exceeds the target value 0.5 and lowers it when the raw KL drops below the floor 0.01.

The training procedure runs CRL pretraining for up to 100 epochs with early stopping on the β-invariant `val_ref_elbo` metric and a patience of 25 epochs. In practice the VAE run hit early stop around epoch 51 and the disentangled run ran the full 100 epochs. Both runs use the same partner-sampling configuration (one partner per stratum, four strata) and the same intervention noise schedule. After CRL pretraining finishes, a separate downstream probe is trained for 50 epochs on the frozen encoder. The probe used here is `linear_fullz`, which reads the full 24-dimensional latent rather than only the labeled `z_type` block; this is the same probe variant in both runs, so probe choice cannot confound the comparison. Class weights for both presence and type heads are computed once on the training split and held fixed for the entire run.

The architecture itself is small enough to fit comfortably on a single GPU at batch size 128. The transformer is two layers deep with four heads and a 64-dimensional model space, taking 32 fused tokens as input. The multiscale frontend uses learned 1-D convolutions at kernel sizes [9, 19, 39, 159] for audio and [9, 19, 39] for seismic, projected into the model space and pooled with a stride of 16 down to the 32-token sequence. The latent is 24-dimensional and is partitioned into named blocks (presence, type, proximity, environment, free) for the VAE objective, or split into a 12-dimensional signal block and a 12-dimensional environment block for the disentangled objective. The contrastive mode would add a small `d_z → d_proj → d_proj` projection MLP on top of the encoder with a temperature of 0.1, but as noted earlier, that mode has not been trained or evaluated, so its hyperparameters are not part of the reported results.

Hyperparameter selection was manual and ablation-driven, not automated. We did not run grid search, random search, or Bayesian optimization. Instead, each candidate change was launched as a single-variable comparison: a new run differs from a chosen reference run in exactly one science variable, and a project rule enforced at launch time blocks runs that change two science variables at once unless the operator writes an explicit override note explaining why the coupling is acceptable. Sweeps are driven by hand-authored YAML files (for example, the frontend-comparison sweep that compared the multiscale frontend against several Morlet variants on the same downstream metrics) or by an in-process Python list of named experiments. The motivation is scientific honesty over throughput: a Bayesian sweep over twenty knobs at once would deliver a single best-numbers run but would not tell us why any one of the twenty knobs mattered. Manual ablations are slower, but they let us attribute each observed gain or loss to a specific change and to write that attribution into the run directory alongside the metrics. The values reported in the table above are the converged settings after this iterative process.

## Training Curves

Two pretraining metrics are tracked through every CRL run: the β-invariant reference ELBO `val_ref_elbo` (lower is better; this is the early-stop signal) and the auxiliary type-F1 `val_aux_type_f1` (higher is better; this is the discriminative proxy on the latent). The first metric measures how well the model can reconstruct the input through the prior; the second measures how well the labeled `type` block of the latent already separates vehicle classes during pretraining, before any downstream probe is trained. Tracking both at the same time reveals tension between the generative and discriminative goals that a single number would hide.

[FIGURE 5: Validation reference ELBO over CRL pretraining epochs for both reported runs (log y-axis). Path: crl-train/saved_crl/slides-figs/fig3_crl_val_ref_elbo.png]

[FIGURE 6: Validation auxiliary type-F1 over CRL pretraining epochs, with best-epoch markers showing where each model's discriminative checkpoint was saved. Path: crl-train/saved_crl/slides-figs/fig4_crl_aux_type_f1.png]

The VAE run improves quickly. `val_ref_elbo` falls below 1.0 by epoch 11 and reaches its minimum of 0.553 at epoch 25. After that, the metric plateaus and even drifts upward, so early stopping triggered at epoch 51 (the patience window of 25 epochs without improvement closed). The auxiliary type-F1 tells a different story: it peaks at 0.607 at epoch 4 and then decays back to 0.385 by epoch 50. The model keeps improving its reconstruction-plus-KL bargain after epoch 4, but it does so by reusing latent capacity that was previously sorted by class. This is exactly the kind of generative-vs-discriminative tension the dual-checkpoint design is built to handle: `crl_best.pth` is saved when `val_ref_elbo` is lowest, while `crl_best_aux_type.pth` is saved when `val_aux_type_f1` is highest, so downstream probes can choose the backbone that fits their task. The disentangled run looks different. `val_ref_elbo` keeps falling for the entire 100 epochs and reaches 0.252 at the last epoch, less than half the VAE's best. `val_aux_type_f1` climbs to 0.695 by epoch 32 and then oscillates around that level. The disentangled model has not converged on the generative metric — it would likely have continued to improve with a longer training budget — but the auxiliary signal it cares about for downstream classification has already saturated.

[FIGURE 7: Validation type-F1 of the linear_fullz downstream probe over its 50 training epochs, on top of the frozen CRL backbone, for both runs. Path: crl-train/saved_crl/slides-figs/fig2_downstream_type_f1_curves.png]

The downstream probe curves expose a few peculiarities worth flagging. Both probes hit their best validation type-F1 at epoch 1 — 0.643 for VAE and 0.646 for disentangled — and then drift down slightly over the remaining 49 epochs. This is not a sign that the probe is broken; the probe sits on a frozen latent, so the type head learns its weights almost immediately and then any further training can only redistribute mass across a fixed feature basis. The presence head behaves differently: under the VAE backbone it climbs steadily and reaches its best validation F1 of 0.865 at epoch 48, but under the disentangled backbone it peaks at 0.843 at epoch 3 and then *unlearns* down to 0.660 by epoch 49. We treat that disentangled-presence collapse as a genuine training-curve characteristic, not a numerical artifact, because the train-set presence loss continues to fall while the validation loss rises. It is most likely a sign that the disentangled latent is splitting presence information across the signal and environment blocks in a way the simple linear probe cannot recover from for very long. Reporting the *best* checkpoint in the figures section is therefore not over-fitting to a single epoch — it is the right summary, since the dual-checkpoint design always picks an epoch by an honest validation metric, never by the test set.

The training curves also tell us about overfitting. During CRL pretraining, the gap between train and validation auxiliary type-F1 is small for the VAE run (about 0.09 over the last ten epochs) and much larger for the disentangled run (about 0.25). That gap is the auxiliary head fitting the training distribution, which is expected because the head reads directly from `μ` and the encoder is being pushed by several losses at once; the gap does not flag a problem with the encoder itself, since the linear-probe downstream numbers from a frozen encoder are essentially tied between the two runs (0.864 vs 0.843 presence F1 and 0.643 vs 0.646 type F1). The disentangled run trades a noisier auxiliary signal for a much better generative score and matches the VAE on downstream F1 at the end. We read the combined picture as: the multiscale frontend is strong enough that either training objective can extract the available signal, and the choice between VAE and disentangled is therefore a choice between different *failure modes* — VAE early-stops cleanly but loses discriminability, disentangled keeps reconstructing well but overfits its auxiliary head — rather than a clean win for either side.

## Performance Evaluation

The capstone ships two distinct models, one per task, because no single (run × probe × checkpoint) combination wins both presence and vehicle-type at the same time. The detection model is run `2026-05-03_05-02-44` (Multiscale + VAE) using the `mlp_ztype` probe loaded from the `crl_best.pth` checkpoint, which is the β-invariant ELBO selector. The vehicle-type model is run `2026-05-03_05-03-14` (Multiscale + Disentangled) using the `linear_signal` probe loaded from the `crl_best_aux_type.pth` checkpoint, which is selected by the auxiliary type-F1 metric during pretraining. Both models are evaluated on the same held-out test split. The presence test split contains 91,325 paired (audio, seismic) one-second windows; the type test split is the subset of 68,083 windows with a vehicle present, since vehicle type is undefined for background windows. Confidence intervals on proportions (precision, recall, specificity, accuracy) use Wilson score intervals; intervals on derived F1 metrics use a non-parametric bootstrap with 1,000 resamples.

The detection model meets the capstone's 0.85 presence-F1 target with statistical clearance: the bootstrap 95% CI [0.8561, 0.8603] sits entirely above 0.85. Recall and precision are roughly balanced (0.841 and 0.876), and specificity at 0.651 reflects a deliberate trade — `mlp_ztype × crl_best` was selected because it maximized presence F1, accepting some false alarms in the negative class so that vehicles are rarely missed. The Matthews correlation coefficient of 0.475 is consistent with that asymmetry. The vehicle-type model falls short of the 0.65 capstone target by roughly four points: macro-F1 is 0.6069 with bootstrap CI [0.6021, 0.6115], and the upper CI bound does not reach 0.65. Per-class behavior is split into two regimes. The two well-supported classes — `medium` (40,569 windows, 60% of vehicle test set) and `heavy` (16,309 windows, 24%) — carry the macro average with F1 of 0.827 and 0.630. The two minority classes — `pedestrian` (6,082 windows) and `light` (5,123 windows) — sit near 0.48 to 0.49 F1 and pull macro-F1 down. Closing the type gap is therefore primarily a minority-class problem, not a global accuracy problem.

The confusion matrices make the dominant error modes concrete. For the type shipping model, the most common confusion is *into* the largest class. Of the 16,309 heavy-class windows, 3,495 (21%) are predicted as medium; of the 5,123 light-class windows, 2,025 (40%) are predicted as either medium or heavy; and of the 6,082 pedestrian windows, 988 (16%) are predicted as light and 1,434 (24%) as heavy. The diagonal of the confusion matrix is dominant on `medium` (recall 0.787) and weakest on `pedestrian` (recall 0.537). This is the classic frozen-linear-probe pathology on a class-imbalanced problem — the probe inherits a "predict medium" prior from the dataset, and rare classes pay the cost. It is a probe limitation rather than an encoder limitation: a deeper probe head, or a class-rebalanced training scheme on top of the frozen encoder, would likely move these numbers without re-running CRL pretraining. For the detection shipping model the dominant error mode is **false negatives on the positive class**, with 10,824 missed vehicle windows (15.9% of all true positives), which is consistent with the capstone preferring the F1-optimal operating point over the high-recall point.

[FIGURE 11: Per-dataset cross-location performance — FOCAL, IOBT, and M3NVC for both shipping models. Path: crl-train/saved_crl/analysis/cross_location_heatmap.pdf]

Splitting the test set by source dataset reveals that neither shipping model is uniform across collection sites. The detection model holds up well on FOCAL (presence F1 0.847, n=32,674) and on M3NVC (presence F1 0.868, n=57,076), but drops to 0.617 on IOBT — a much smaller test slice (n=1,575) where the smaller sample size widens the confidence band considerably. The type model is more uniformly weak across sites: macro-F1 is 0.437 on FOCAL, 0.412 on M3NVC, and 0.307 on IOBT, with the same caveat that IOBT's small slice means the worst-site number is also the noisiest one. Either way, the capstone's worst-case-per-site target of 0.65 type-F1 is not yet met on any individual collection site. The pattern says the vehicle-type encoder generalizes about as well as the data lets it, and the remaining gap is concentrated in two places — the smaller IOBT slice where statistical confidence is low, and the minority-class supports across all three sites. Closing those gaps is the next round of work; the contrastive training mode (described earlier but not yet trained) is on the roadmap because contrastive objectives are known to give better minority-class structure under imbalanced supervision than reconstruction-based ones.

## Ablation Studies

Every ablation in this paper is a strict one-variable comparison: two runs that differ in exactly one science variable, with every other knob held identical. The pairing is automated by a tool that enforces single-variable discipline at analysis time — given a candidate variant and a reference run, it refuses to produce a delta if more than one tracked axis has changed unless the operator has written an explicit override note explaining why the coupling is acceptable. This rule is the same one used at run-launch time during training, so the ablation table cannot drift away from the experimental record. The deltas reported below are therefore attributable to a single change rather than to a tangle of co-varying knobs.

[TABLE 5: Ablation pairs that meet the one-variable rule, with deltas on the headline metrics. Source: crl-train/saved_crl/analysis/ablations.md and cross_location.md. Δ = variant − baseline; positive Δ means the variant won.]

------------------------------------------------------------------------------------------------------
 Axis                  Baseline → Variant                   Δpres_f1   Δtype_f1   Δref_elbo
------------------------------------------------------------------------------------------------------
 training_mode         disentangled → vae                   +0.023     −0.031     +0.300
 frontend_type         morlet_per_sensor → multiscale       +0.055     +0.115     −3.91
 prior_type            standard → conditional (iVAE)        — (no paired runs) —
 training_mode         vae → contrastive                    — (variant not yet trained) —
------------------------------------------------------------------------------------------------------

The training_mode ablation pairs the two shipping runs against each other. Going from the disentangled objective to the VAE objective gains 0.023 on presence F1 but loses 0.031 on vehicle-type macro-F1 and degrades the β-invariant reference ELBO by 0.300. The split is exactly what the loss anatomy in the Training Objective section predicts: the VAE objective tightens reconstruction but does not enforce a signal-versus-environment split, so the type information ends up less localized and the type probe loses ground; the disentangled objective gives a much stronger generative bargain (lower ref-ELBO) and a slightly stronger type signal at the cost of presence accuracy. The capstone's choice to ship both models at once — VAE for detection, disentangled for type — is the natural answer to this trade-off, since neither objective wins both heads.

The frontend_type ablation compares the multiscale frontend used in the shipping models against the older morlet_per_sensor frontend on the same backbone, prior, and training mode. Switching from a fixed Morlet wavelet bank to learned 1-D convolutions improves presence F1 by 0.055 and vehicle-type macro F1 by 0.115, and lowers `val_ref_elbo` by 3.91 — a much larger generative gap than the training-mode delta. The cross-location numbers reinforce the same conclusion: at the worst site the morlet frontend reaches only 0.308 type macro-F1, while the multiscale frontend lifts that floor to 0.436 (disentangled) or 0.455 (VAE). This is the single largest ablation effect we observed and is the reason the paper reports multiscale-only results going forward; the Morlet variants are documented in the run history but are not part of the shipping comparison.

Two further axes are knobs in the framework but do not yet have paired runs in the record, so we cannot quote deltas. The first is `prior_type`: the framework supports a conditional iVAE-style prior in addition to the standard Gaussian, but no run has been trained against the conditional prior under otherwise-identical settings, so we have nothing to compare against. The second is `training_mode → contrastive`: the NT-Xent contrastive mode is implemented and shares the same encoder, frontend, and data pipeline, but as noted earlier in the Training Objective section it has not been trained or evaluated. We deliberately did not back-fill these deltas with weaker pairs that violate the one-variable rule, because a delta that conflates two changes is worse than no number at all — it can mislead a future reader into attributing an effect to the wrong cause. Closing both gaps is the next round of work, and when those runs land they will appear as new rows in the ablation table under the same single-variable discipline applied here.

## Appendix

### Glossary

**ADC counts** — raw integer samples written by the analog-to-digital converter on each sensor before any normalization.

**Auxiliary head** — a small classifier (presence or vehicle-type) that reads the latent posterior mean μ during CRL pretraining and is trained jointly with the encoder so the labeled latent blocks stay tied to label structure. Distinct from the downstream probe, which is trained later on the frozen encoder.

**β (beta)** — the coefficient on the KL term in the VAE loss. Annealed during training from 0 toward 1 to prevent latent collapse early on; the β-invariant reference ELBO `val_ref_elbo` is evaluated at β = 1 so checkpoint selection does not drift with the schedule.

**Bootstrap CI** — confidence interval estimated by resampling the test set with replacement (1,000 resamples in this paper) and taking percentiles of the metric distribution. Used here for derived metrics like F1 where Wilson intervals do not apply.

**Capstone target** — 0.85 presence F1 and 0.65 vehicle-type macro F1, both per-site. The "ship metric" is the worst-case-per-site F1, so a model passing the capstone target on the pooled test set may still fail the per-site target.

**Checkpoint, dual** — the framework saves two CRL checkpoints per run: `crl_best.pth` selected by `val_ref_elbo` (generative anchor) and `crl_best_aux_type.pth` selected by `val_aux_type_f1` (discriminative anchor). Downstream consumers pick whichever serves their task.

**Contrastive mode** — NT-Xent training objective using stratified anchor/partner pairs. Implemented in the framework but not trained or evaluated in this paper.

**CRL (Causal Representation Learning)** — the family of pretraining objectives used here. The latent space is partitioned into named blocks (presence, type, proximity, environment, free) so that downstream consumers can target the block their task needs.

**Disentangled mode** — VAE variant that splits the latent into a signal block and an environment block, with explicit alignment, temporal-stability, and intervention-invariance losses driving the split. Used as the shipping model for vehicle type.

**Frozen-linear-probe pathology** — the phenomenon where a linear classifier on top of a fixed feature space inherits the class prior of the dataset and over-predicts the dominant class. Visible in this paper as confusion mass collected in the medium column.

**FOCAL / IOBT** — two collection sites of the MOD_vehicle dataset. Different locations and vehicle fleets, but one shared sensor setup and file format.

**Intervention noise** — one of seven generators (white, brown, pink, green band-limited, low-frequency sinusoid, high-frequency chirp, bird-chirp-like) added to anchor windows at 20% RMS. Drives the disentangled mode's signal-invariance loss; available to other modes as augmentation.

**`linear_fullz` probe** — downstream classifier that reads the full 24-dimensional latent rather than a single named block. Used early in this paper for fair backbone comparison; not a shipping probe.

**`mlp_ztype` probe** — downstream MLP classifier on the labeled `z_type` block of the latent. Used as the shipping presence head for the VAE run.

**`linear_signal` probe** — downstream linear classifier on the disentangled-mode signal block. Used as the shipping type head for the disentangled run.

**MOD_vehicle dataset** — multi-sensor vehicle-pass dataset collected by ARL and the University of Illinois Urbana-Champaign with Raspberry Shake nodes. Comprises the FOCAL and IOBT sites in this paper.

**Multiscale frontend** — three or four parallel learned 1-D convolutions at kernel sizes [9, 19, 39, 159] for audio and [9, 19, 39] for seismic, projected via 1×1 conv to the model dimension and pooled with stride 16 to a 32-token sequence.

**M3NVC** — open research dataset contributing background scenes and four passenger vehicles. Different sensor rig from MOD_vehicle; resampled at load time to match.

**Reference ELBO (`val_ref_elbo`)** — `val_recon + val_raw_kl` evaluated at β = 1. Epoch-invariant under the β anneal, so it is the early-stop signal and the `crl_best.pth` selector.

**Ship metric** — the worst-case macro F1 across collection sites. Single number that summarizes cross-location robustness.

**Single-variable rule** — project policy that an experiment may differ from its reference run in exactly one science variable. Enforced at run-launch time and at ablation-analysis time. Operators can override with a documented `coupled_change_note.txt`.

**Stratified pair sampling** — every anchor window is paired with one partner from each of four strata (consecutive same-recording, same-type different-recording, different-type, different-dataset). Same data plumbing serves the VAE intervention-matching head, the disentangled temporal-stability loss, and the contrastive NT-Xent objective.

**Wilson score interval** — confidence interval for a binomial proportion. Used here for recall, precision, specificity, and accuracy — i.e., metrics that are direct ratios over the test set.

## Reflection

The biggest lesson from this project was that scientific discipline pays for itself in time saved later. The one-variable-at-a-time rule felt slow at first, because it meant launching a new run for every single knob instead of stacking changes; but every time we broke the rule we paid for it twice — once when the result was hard to interpret and again when we had to re-run the experiment with the changes split apart. We would carry that habit straight into a future project. A related lesson was about choosing what *not* to keep: we tried several frontends in parallel early on (multiscale, fixed Morlet, learnable Morlet, fused variants), and although the ablation table showed multiscale won decisively, we kept the Morlet branches alive in the codebase and figures longer than we should have, which made the comparison harder to read without changing the conclusion. Next time we would set a clearer criterion up front for when an exploratory branch becomes a kept feature versus a documented dead end. Working as a team forced us to write our reasoning in places teammates could find it; the three artifacts that helped most were the per-run `coupled_change_note.txt` for any knowingly-broken single-variable rule, a colocated figure-builder script next to every figure in the paper, and a short append-only memory of project-level decisions that did not belong in code (such as *why* the type-shipping run uses the auxiliary type-F1 checkpoint rather than the ELBO checkpoint). None of these artifacts were large, but together they made review and hand-off much cheaper, and we would build that habit on day one rather than midway through. The decision-making process for the paper itself was iterative and grounded in the runs on disk: we did not start with an outline, we started with the latest `meta.json` and `eval_report.json` files and let the numbers tell us what we could honestly say, re-checking the underlying CSV or JSON whenever we were tempted to round up or phrase a result more strongly than the data supported. The clearest example was when we initially reported Performance Evaluation numbers using a `linear_fullz` probe that was common to both runs — clean for a head-to-head, but not the per-task shipping probes the project actually deploys. Catching that and rewriting the section against the correct probes was the moment the paper became a description of what the project ships rather than a description of a clean ablation, and next time we would design that probe-choice check into the writing process from the start.
